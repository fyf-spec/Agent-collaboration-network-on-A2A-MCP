1

Motivation

Networking project requirements ask the system to run services, communicate across processes, log protocols, tolerate failures, and visualize behavior.

  • A2A requirement: Coordinator dispatches domain agents through explicit network messages.
  • MCP requirement: Agents invoke external tools through JSON-RPC MCP servers.
Goal: a discoverable, observable, fault-aware travel-planning network.
2

Problems

  • Service discovery is fragile: registry state, ports, and health can change while a task is being routed.
  • TCP is a byte stream: raw TCP provides no inherent boundaries, necessitating custom framing to parse structured A2A tasks and status codes.
  • Tool calls amplify traffic: repeated weather and short-window duplicate calls hit MCP servers unnecessarily.
  • Partial failures are normal: missing agents or slow MCP servers should degrade the answer, not break the whole DAG.
3

Contributions

  • Intelligent Coordinator that decomposes travel requests, dispatches sub-tasks to domain agents via a dependency-aware DAG, and merge their results.
  • Primary/Backup Registry Centers that enable dynamic agent discovery through heartbeat monitoring instead of hard-coded endpoints.
  • Five specialized agents (weather, traffic, attraction, hotel, and packing) that wrap external MCP tools into task-specific planning advice.
  • A2A/TCP protocol with explicit message framing (length-prefix), state tracking (ACK/RESULT/ERROR), and persistent channel management.
  • MCP Gateway that centralizes MCP-side cache reuse, duplicate coalescing, backpressure, circuit breaking, and /metrics observability.
  • MCP tools that expose services as HTTP/JSON-RPC endpoints behind Gateway.
  • Fault-tolerance designs handle registry failover, capability misses, MCP timeout, retry/no-LLM fallback, and explicit degraded answers.
4

System Architecture main design

User → Coordinator → Registry → A2A/TCP → Agent → MCP Gateway → JSON-RPC → MCP Server
5

Protocol Advantages & Fault Tolerance

Protocol advantage: A2A replaces HTTP POST on the agent data plane with a lightweight TCP frame: a 4-byte length header plus JSON payload.
• cuts HTTP header and parsing overhead• keeps a reusable channel for frequent task dispatch• makes message boundaries explicit in the TCP byte stream

Fault-tolerance design: each failure has a local recovery boundary: registry failover, agent degradation, Gateway fail-fast, and service power-off recovery with visible logs.

6

Governance Efficiency

Gateway role: control MCP traffic before it reaches tool servers, preserve JSON-RPC semantics, expose /metrics, and turn repeated or failing tool calls into measurable governed behavior.

N → 1
Upstream call reduction via caching and coalescing
≤ 2
Upstream concurrent load under Gateway backpressure
16.4x
Faster circuit-open responses than direct slow timeout
  • Cache reuse: metric is upstream MCP calls under repeated identical requests; Gateway turns repeats into cache hits and keeps the load near 1 call.
  • Duplicate coalescing: metric is upstream calls under concurrent duplicate requests; Gateway keeps one leader call while waiters share the result.
  • Backpressure: metric is accepted vs. rate-limited requests; Gateway bounds in-flight tool work instead of letting bursts hit MCP directly.
  • Circuit breaker: metric is latency after timeout failures; Gateway opens the circuit and converts later slow calls into fast error responses.
Benchmark setting: local Weather MCP, 3 repeats; repeated requests 10-100, duplicate/unique bursts 1-40/2-40 concurrent requests, and 8 sequential timeout calls. All metrics are measured locally.

Travel QA Example and Failure Semantics

Complete flow: "Plan a 3-day budget trip from Shanghai to Beijing; prefer subway; must visit the Forbidden City."
Usersubmits the travel-planning request to the Coordinator.
Coordinatorqueries Registry for agent endpoints; Registry sends back agent health to Coordinator.
Registryprimary lookup can fail; backup discovery returns the available agent endpoints.
Agentsexecute the DAG through A2A/TCP and call MCP tools through the Gateway.
Gatewaycaches repeats, coalesces duplicates, limits bursts, and fails fast on slow MCP servers.
Answermerges available results into an itinerary and marks unavailable tools explicitly.
  • Failure semantics: registry failover, capability miss, and MCP timeout all stay visible in logs and final context.
  • Partial answer: usable nodes return first; missing tool results are reported as explicit degraded fields.
